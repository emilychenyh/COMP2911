public class RectanglePlus implements Relatable {
    // fields, constructors

    public int isLargerThan(Relatable other) {

        RectanglePlus otherRect = (RectanglePlus)other;

        if (this.getArea() < otherRect.getArea()) {
            return -1;
        } else if (this.getArea > otherRect.getArea()) {

        }

public Object findLargest(Object object1, Object object2) {

    Relatable obj1 = (Relatable)object1;

Inheritance Theory

Subclass
a class that is derived from another class (extends keyword)
inherits all public and protected members (fields, methods, and nested classes), note constructors are not members
aka, derived class, extended class, child class
Superclass
class from which the subclass is derived
aka, base class, parent class
Single Inheritance
every class has one and only one direct superclass
every class is implicitly a subclass of Object

Overriding and Hiding Methods

  • overriding — instance method in subclass with same signature as subclass
    • called from superclass => get overriding subclass method
    • inherit method that is “close enough”, then modify behavior as needed
    • covariant return type — can return subtype of type returned by overriden method in superclass
    • use @Override
  • hiding — static method in subclass with same signature as superclass
    • called from superclass => get superclass static method
    • called from subclass => get subclass static method

  • typecasting is covariant
// MountainBike ⇒ Bike ⇒ Object
Object obj = new MountainBike();    // covariance
MountainBike myBike = (MountainBike)obj;    // explicit casting
  • [X] multiple inheritance of state — ability to inherit fields from multiple classes
    • problem — methods/constructors of different superclasses instantiate same field
    • => solution — interfaces do not contain fields
  • [~X] multiple inheritance of implementation — ability to inherit method definitions from multiple classes
    • problem — name conflicts and ambiguity
    • problem — class can implement multiple interfaces, with default methods of the same name
    • => solution — Java compiler has rules to determine which it uses
  • [√] multiple inheritance of type — ability of a class to implement more than one interface
    • multiple types — its own class + all the interfaces it implements
      • using an interface as a type

Liskov Substitution Principle

Liskov Substitution Principle
derived subtypes must be completely substitutable for their base types
functions that use base types can be reused with impunity, derived types can be changed with impunity
functions that use pointers or references to base classes must be able to use objects of derived classes without knowing it
Dynamic Binding
consequence of inheritance
Open-Closed principle
well designed code can be extended without modification
new features are added by adding new code rather than changing old, already working code

Mechanisms of Open-Closed principle

  • abstraction
  • polymorphism
  • => inheritance

Example of Violation of LSP

  • Rectangle base class with derived Square class
  • “is-a” relationship holds => logical to subtype
  • problems
    • Square doesn’t need both height and width
    • can modify setters to change them together
    • BUT passing a Square to a function that assumed Rectangle will have assert failures when changing height then width
    • this is not an ISA relationship
      • extrinsic public behavior of Square and Rectangle differ
  • => validity of a model expressed in terms of clients
    • have to anticipate assumptions made
  • solution — Design by Contract
    • specify preconditions and postconditions
    • for derivatives, may only replace
      • precondition by a weaker one
      • postcondition by a stronger one

4 Common Pitfalls

Equals may seem to work fine, but a lot of problems come up when you put objects in collections (i.e. hash sets). DO the following:

  1. Define equals with the right signature

    • need to match the signature exactly, use Object not subclass, then typecast
    • otherwise not actually overriding
    @Override
public boolean equals(Object other) {
    boolean result = false;
    if (other instanceof Point) {
        // typecast if is subclass that you want
        Point that = (Point) other;
        // NOW, see if the two are equal
        result = (this.getX() == that.getX()) && (this.getY() == that.getY()) && this.canEqual(that);
    }
    return result;
}

  2. Change hashCode whenever you change equals

    • problem when using HashSet
    • hashCode may only depend on fields that equals depends on 
      // i.e. if equals tests for x and y of Point above
// can use x and y fields in hashCode
@Override
public int hashCode() {
    return (41 * (41 + getX()) + getY());
}

  3. Define equals and hashcode in terms of immutable fields

    • make sure fields used are final
      • otherwise equality and hash code changes with fields
    private final int x;
private final int y;

  4. Define equals as an equivalence relation using canEqual

    • add canEqual method to every class that redefines equals
      • objects of a class != objects of superclass
    • check for canEqual when checking for equality
    public boolean canEqual(Object other) {
    // replace Point with whatever subclass you are in
    return (other instanceof Point);
}
    • on non-null objects x, y, z
      • reflexive: x.equals(x) returns true
      • symmetric: x.equals(y) iff y.equals(z)
      • transitive: x.equals(y) and y.equals(z) => x.equals(z)
    • more complicated once subclassing
      • problem => symmetric and transitive breaks
      • solution => redefine canEqual method
    • but violates LSP
      • should be able to substitute subclass into superclass
      • but LSP doesn’t require subclass behave identically, just fulfill contract of superclass

Applying “Design by Contract”

Components of Design by Contract

  • set of methodological principles to produce correct and robust software
    • correct — returns accurate and reliable results
    • robust —ensure software continues to operate, even if it results in inaccuracues
  • exception-handling mechanism
  • understanding of inheritance and associated techniques (redeclaration, polymorphism, dynamic binding) via subcontract

Defensive Programming

  • protect software against all possible events
  • increases software complexity ⇒ dec software quality
  • uses assertions to document pre and post
    • disable in production so app doesn’t crash

Components of Design by Contract

  • precondition — requirements that any call must satisfy if the call is to be correct
  • postcondition — properties that are ensured in return by the execution of the call
  • class invariants — property that applies to all instances of the class, transcending particular routines
    • all procedures must satisfy the invariant
    • incorporated with pre and post of all methods

Law of Demeter

Law of Demeter (LoD)
each unit should have only limited knowledge about other units; only units “closely” related to the current unit
each unit should only talk to its friends; don’t talk to strangers
only talk to your immediate friends
aka principle of least knowledge
  • specific case of loose coupling
    • assume as little as possible about the structure / properties of anything else
  • Law of Demeter for Functions/Methods (LoD-F)
    • specific for OOP
    • A shouldn’t reach through object B to access an object C
    • solutions
      • modify B’s interface
      • A should directly reference C
    • “use only one dot”
      • breaks law — a.b.Method()
      • complies — a.Method()
        • want dog to walk ⇒ command the dog ⇒ dog commands legs
        • don’t command the legs directly
  • function that requires a method m of an object o may only invoke the following
    • o itself
    • m’s parameters
    • any objects created/ instantiated within m
    • o’s direct component objects
    • a global variable, accessible by o, in the scope of m
    • NOT — objects returned by m
  • advantages
    • increase maintainability and adaptability
    • containers changed without reworking callers
  • disadvantages
    • many wrapper methods ⇒ time & space overhead
